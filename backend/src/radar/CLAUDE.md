# backend/src/radar/ — the External Content Radar (ARC-16 / EXT)

**Purpose.** Agenda-driven, grounded discovery of external items (events / news /
research) the org could comment on — the "second content well". Highest
reputation-risk surface (R-4). Implements the behaviour spec **EXTS**
(`.spec/specs/ext-external-radar.yaml`).

| File | Role | Spec |
|---|---|---|
| `guard.ts` | The deterministic R-4 source guard: `provenanceBound` + `applyR4Guard` (keep only provenance-bound + dereferenceable candidates) | `@implements EXTS-1` |
| `store.ts` | DM-8 ExternalItem persistence: `persistCandidates`, `listDiscoveries` (pull-only feed), `savedPool` (GEN-1), `setDisposition` | `@implements EXTS-1/EXTS-5` |
| `index.ts` | `createRadar` facade — `discoverRun` (agenda → grounded search → R-4 guard → persist), `discoveries`, `savedPool`, `triage` (→ Memory) | `@implements EXTS-1/EXTS-5` |

**The R-4 guard (EXTS-1, load-bearing).** Discovery runs Gemini + Google-Search
grounding (`port.groundedSearch`, IG-3); the port returns candidates **plus the
grounding `sources`** (the URLs the retrieval actually cited). `applyR4Guard`
keeps a candidate ONLY if its URL is provenance-bound (in `sources` — not
model-invented) AND dereferenceable (an injected HTTP check; faked in tests).
Hallucinated + dead links are dropped before the mandatory GR-5 citation. LRN-20:
the guard is deterministic structure/policy; candidate *relevance* is the LLM/search
judgment (held by the rationale + the OPS-1 precision review), never claimed
deterministic.

**Discoveries (EXTS-5).** The feed is pull-only — never badged/counted/nudged.
Each triage disposition writes Memory (`discovery-triage`, MEMS-1); `saved-for-later`
joins the pool the planner draws from (GEN-1).

**Deferred.** EXTS-2 external drafts are the content engine's (GENS hands a
worth-a-post candidate to draft); EXTS-3 the permanent TL1 cap is AUTS-2; EXTS-4
structured-feed source depth is P1. v0 geography is a broad scope constant
(org-specific geography from a Memory geo-fact is a refinement); refining the
TOPS-2 per-topic research strategies from triage feedback is the deeper loop.

**Boundaries.** External search goes through the LLM/search port (`../ports/llm.ts`,
ADR-0003), never a vendor SDK. Org-scoped everywhere (ACC-3). The agenda is
injected (`agendaFor`) so the module stays decoupled from TOPS.

**Gotcha (keyed grounding — LRN-28).** The Vertex `groundedSearch` adapter wires
`vertex.tools.googleSearch` + reads `result.sources`; it typechecks + runs but the
keyed smoke returned **0 grounding sources** — grounding did not fire in this
two-pass shape. The deterministic tier (R-4 guard, persistence, feed, triage) is
green and the keyed radar quality is dormant, so nothing ships as falsely-working;
verifying the Vertex grounding wiring (likely `providerMetadata.google.groundingMetadata`)
is a tracked follow-up before the keyed radar is relied on.
