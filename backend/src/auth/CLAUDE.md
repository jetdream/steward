# backend/src/auth/ — the BetterAuth identity mechanism (ARC-10)

**Purpose.** The identity MECHANISM (ACCS-1, ADR-0006, SEC-7). There is no
bespoke identity store — BetterAuth's `organization` + `admin` plugins ARE the
User/Org/Membership model (DM-15/16/1), and `emailOTP` backs the dev sign-in.

| File | Role | Spec |
|---|---|---|
| `auth.ts` | `createAuth(db)` — the BetterAuth instance (Drizzle adapter over `@shared/db/auth-schema`, Google social provider, org+admin+emailOTP plugins, trusted origins) + `devOtpStore` | `@implements ACCS-1 v1` |
| `auth.config.ts` | BetterAuth CLI config (schema generation) | — |

**Sign-in (SEC-7).** Google is the **production** provider (credentials from
env). The **dev** flow is email-only: `emailOTP` captures the code into
`devOtpStore` (no mail sent) and `routers/auth.ts::devLogin` consumes it — dev
login is refused when `NODE_ENV=production`.

**Boundaries.**
- The tRPC surface (devLogin / me / logout, org list/create/switch, membership
  lifecycle) is `../routers/{auth,org}.ts` — this folder is config only.
- Server-side org confinement (ACCS-2) is `orgProcedure` in `../trpc.ts`.
- The Steward single-owner invariant (ACCS-3) is `../accounts/` policy (BetterAuth
  does not enforce it).
- **Org (DM-1) IS the BetterAuth `organization`** (DEC-39) — Steward org fields are
  added via the org plugin's `additionalFields`, never a separate profile table.

**Gotcha.** `auth.ts` imports the schema via the `@steward/shared/db/auth-schema`
package export (the loader/CLI path), not the `@shared/db/*` alias domain modules
use — BetterAuth's adapter + its CLI both resolve it that way.
