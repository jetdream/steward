# backend/src/topics/ — the editorial-agenda engine (ARC-26 / TOP)

**Purpose.** The system owns the editorial AGENDA (DEC-23): *what the org talks
about*. It proactively identifies content topics grounded in Memory and exposes
the active-topic set as the agenda — the single source the planner (GEN-1),
Strategy section (a) (STRS-1), and the Radar (EXT-1) read (TOP-4, one source, no
second copy).

**TOPS-1 slice (this module today).** The `identify-topics` Skill on the
ARC-27/PIPE-4 substrate, evaluated day-one (ADR-0010).

| File | Role | Spec |
|---|---|---|
| `identify.ts` | `deriveTopics` (Skill run + the guard, pure) · `applyEvidenceGuard` (deterministic) · `identifyForOrg` (MEMS listGrounding → identify seam) | `@implements TOPS-1 v1` |
| `store.ts` | `persistTopics` (DM-13, active/system-derived, dedup by `topicKey`) · `getAgenda` / `activeTopics` · `topicKey` (canonical) | `@implements TOPS-1, TOPS-4 v1` |
| `index.ts` | `createTopics({db, memory, port})` facade — `identify` + `getAgenda` | — |

**Grounding + the guard (LRN-20).** Identification is a grounded LLM Skill:
`memory.listGrounding` supplies the org's active grounding entries (broad, not a
similarity slice), the model proposes topics citing the Memory entry ids that
ground them, and the **deterministic evidence guard** (`applyEvidenceGuard`) drops
any topic whose evidence does not resolve into that set — never persisted. Set
membership, no content heuristic. Rationale QUALITY (is the "why it fits"
grounded vs plausibly-fabricated over real ids) is the keyed catch-rate tier
(dormant on the keyless stub, like the guardrail judge) — a follow-on rationale
judge, not built here.

**Auto-drafted agenda (VAL-6).** Identified topics land ACTIVE, system-derived —
the agenda is drafted first, never a blank page. `getAgenda` returns the active
set.

**Deferred:** TOPS-2 research strategy (Radar), TOPS-3 proposals (PRO-4 budget +
the XP-1 card), the `editAgenda` edit half of TOPS-4, and topic EVOLUTION
(re-describe/retire via DM-13 supersession) — a re-run dedups by `topicKey`
rather than diffing the set.

**Gotcha.** Any prompt/model change to the `identify-topics` Skill bumps the
harness-manifest hash → the eval gate requires a fresh passing run.
