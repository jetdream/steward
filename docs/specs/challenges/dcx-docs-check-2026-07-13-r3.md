- [severity: high] Markdown-frontmatter parse errors are silently discarded, so a duplicate `status:` key in an ADR's frontmatter last-wins and passes the DCX-11 approval gate — direct contradiction of DCX-4 v4's unqualified claim "YAML-subset parse errors are errors — including duplicate keys at the same level, which plain YAML would silently last-win" — survives scrutiny because it is the exact silent-last-win corruption mode round 2's high finding was fixed for, resurfacing on the channel that feeds gate-relevant metadata (ADR status drives DCX-11; verified live: `status: proposed` + duplicate `status: accepted` frontmatter yields zero parse errors and satisfies a `constrained-by: [ADR-0001]` check on an approved spec); "not schema-validated" in DCX-4 does not refute it, since parse validity and schema validity are distinct clauses of the same item — fix: in `buildGraph` (scripts/lib/docs-graph.mjs:155) propagate `parseYamlSubset(...).errors` from frontmatter into graph errors (with the +1 line offset), or, if frontmatter is deliberately lenient, say so explicitly in DCX-4 and make DCX-11 reject ADRs whose frontmatter failed to parse.
- [severity: medium] Defining an item under an unregistered, unowned prefix — or claiming an unregistered prefix in a file's `prefix:` key — produces zero errors, and the phantom item enters the `--json` canonical registry (verified: `ZZZ-1` in goals.yaml items and `prefix: [G, ZZZ]` both exit 0) — survives because docs/CLAUDE.md promises "docs-check rejects unregistered ones [prefixes] before first use" and the spec's purpose claims to enforce prefix ownership, but DCX-2/DCX-3 as written only cover owned-prefix definitions and references — fix: error on any items-map definition or top-level prefix claim whose prefix is absent from the registry, and extend DCX-2's rule text accordingly.

- [severity: low] Docs files with extensions outside `.{yaml,md}` (e.g. `.yml`) are invisible to every check (verified: `docs/product/stray.yml` with a dangling ref exits 0) — technically spec-conforming since DCX scopes itself to yaml|md, so non-blocking — fix: a folder-hygiene clause (DCX-7) erroring on unexpected extensions under docs/.
